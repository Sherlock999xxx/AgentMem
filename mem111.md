# mem111: AgentMem 当前记忆平台评估与 ProactiveAgent 改造增量

## 1. 本轮目标

本轮聚焦 `task-1772351678-b1f8`，补齐 `agent-mem-proactive` 的核心门面设计，并把这一阶段对 AgentMem 记忆平台的评价、差距与后续计划沉淀为中文文档，作为后续集成与迁移工作的输入。

## 2. 当前代码现状

### 2.1 已完成的文件中心化改造

- `agent-mem-resource` 已提供资源抽象，补上了“先有资源，再提取记忆”的入口。
- `agent-mem-category` 已实现层级类别树，能够承接 memU 风格的文件夹式组织。
- `agent-mem-extraction` 已建立提取管道框架，具备从资源到记忆项的中间层。
- `agent-mem-proactive` 已具备调度器、任务模型、执行器六大组件，是主动整理能力的基础。

### 2.2 `agent-mem-proactive` 真实实现评价

优点：

- 已有 `ProactiveTask`、`TaskSchedule`、`SchedulerState`、`TaskResult` 等核心模型，基础数据面完整。
- 已有 `TaskScheduler` 具体实现，支持定时轮询、即时执行、启停、启用/禁用任务。
- 已有六类执行器：自动分类、去重合并、摘要生成、索引优化、资源归档、健康检查。
- 已有较完整的单元测试覆盖模型、调度器与执行器，说明 crate 具备独立演进能力。

不足：

- 之前缺少真正的 `ProactiveAgent` 门面，外部只能直接操作调度器，无法完成“统一初始化 + 注册默认执行器 + 按配置装配任务”的启动流程。
- 默认任务集不完整，只覆盖了 4 个任务，和设计目标中的 6 个内置后台任务不一致。
- 多个执行器仍是占位实现，核心流程里保留了 `TODO`，说明目前是“架构壳子已成型、业务闭环尚未接通”。
- `TaskScheduleConfig.priority` 还没有进入实际调度决策，当前调度仍偏静态。
- 事件驱动触发仍停留在 `TaskSchedule::event()` 标记层，没有真正与 Event Bus 或资源摄取流程联动。

结论：

当前 AgentMem 已经从“没有主动整理层”提升到“有独立主动代理 crate，但还没有完整启动编排与真实业务执行”的阶段。它比最初规划前进了一大步，但距离可持续运行的 24/7 Proactive Memory Agent 还有明显工程缺口。

## 3. 本轮架构增量

### 3.1 新增 `ProactiveAgent` 门面

本轮新增高层 `ProactiveAgent`：

- 持有 `ProactiveConfig`
- 持有底层 `TaskScheduler`
- 统一注册 6 个默认执行器
- 支持按配置装配任务
- 支持默认任务启动
- 暴露 `initialize / start / stop / list_tasks / run_task_now / stats / state`

这让 `agent-mem-proactive` 从“零散组件集合”变成了“可直接启动的子系统”。

### 3.2 默认任务集合对齐设计目标

默认调度任务已扩展为 6 个：

- `AutoCategorize`
- `DedupeMerge`
- `GenerateSummaries`
- `IndexOptimization`
- `ResourceArchival`
- `HealthCheck`

这与 `PROMPT.md` 和 `todo3.md` 中定义的主动代理目标保持一致，避免默认启动后缺少索引优化和资源归档两类关键后台任务。

### 3.3 当前阶段的架构判断

可以认为主动代理现在具备了“三层结构”：

1. 门面层：`ProactiveAgent`
2. 调度层：`TaskScheduler`
3. 执行层：六类 `TaskExecutor`

这种结构是合理的，因为：

- 门面层负责启动编排和配置绑定
- 调度层负责生命周期与执行时序
- 执行层负责具体后台任务逻辑

这比让外部系统直接拼接 scheduler + executors 更稳，也更利于未来接入 server、SDK 与事件系统。

## 4. 与顶级记忆平台思路的对比

结合前期分析，AgentMem 当前和更成熟的记忆平台相比，差距主要还在这里：

- 与 memU 相比：资源抽象、类别层级、主动代理三大方向已经开工，但“资源 -> 提取 -> 分类 -> 检索 -> 主动整理”还没完全串成闭环。
- 与 mem0 / Zep 这一类工程化平台相比：AgentMem 的模块数量更多、目标更大，但端到端稳定流程、默认产品化体验、托管式主动维护能力还不够强。
- 与 MIRIX / MemVid 这类偏长期记忆或压缩存储方向的项目相比：AgentMem 已有丰富后端和搜索引擎基础，但主动整理结果如何反馈给检索质量，还缺少量化指标闭环。

换句话说，AgentMem 当前的优势是“基础设施宽、模块多、改造方向正确”；当前的短板是“主链路还未完全打通，后台主动能力还没有真实接到资源和类别系统”。

## 5. 后续改造计划

建议把主动代理后续工作分成四步：

### 第一步：接通真实数据面

- `AutoCategorizeExecutor` 对接 `agent-mem-category` 与 `agent-mem-extraction`
- `DedupeMergeExecutor` 对接向量搜索与去重策略
- `GenerateSummariesExecutor` 对接类别摘要生成器

目标：从占位返回值升级为真实处理结果。

### 第二步：接入事件总线

- 让 `TaskSchedule::event()` 接入 `agent-mem-event-bus`
- 在资源挂载、提取完成、新记忆写入后触发主动任务

目标：把“事件驱动”从模型定义变成真实触发机制。

### 第三步：引入调度策略

- 使用 `priority`、资源预算、批处理窗口控制执行顺序
- 为 CPU/内存预算做全局限流
- 增加失败重试和熔断策略

目标：避免主动代理失控，满足 `<5% CPU overhead` 目标。

### 第四步：对外暴露统一入口

- 在 `agent-mem` 统一 API 中暴露 Proactive 配置
- 在 server / Python SDK / Client SDK 中支持开启、关闭、查看后台任务
- 为 SDK 迁移准备兼容层

目标：让主动代理成为平台能力，而不是单独 crate 内部能力。

## 6. 对当前平台的总评价

总体评价：`7.5 / 10`

原因：

- 架构方向正确，且与文件中心化记忆平台演进方向一致。
- 核心 crate 已经分层完成，基础模型质量不错。
- 但主动代理、提取管道、类别系统与统一 API 之间仍有“最后 20% 的集成鸿沟”。

如果后续优先完成 ProactiveAgent 与现有 agents / SDK 的集成，AgentMem 才能真正从“高性能记忆引擎”升级成“以记忆为核心、能主动整理和持续演化的平台”。
