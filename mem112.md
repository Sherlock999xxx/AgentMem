# mem112：AgentMem 面向 Claude Code 类文件中心 AI 记忆平台的总评、外部理念对照与重构计划

> 更新时间：2026-03-19
> 范围：基于当前仓库代码现状、对 `OpnViking` 公开资料的搜索、Anthropic 官方 Claude Code 记忆机制、以及 Letta 的文件系统式记忆路线

## 1. 执行结论

这次更新后的核心结论比上一版更明确：

**AgentMem 目前不是一个已经成型的“Claude Code 类、以文件为核心的 AI 记忆平台”。**

更准确地说，它现在是：

1. 一个以 `Memory / MemoryType / Search` 为主叙事的通用 AI 记忆平台；
2. 同时已经长出了 `resource + category + extraction + proactive` 四条 file-centric 支线；
3. 但这些 file-centric 能力还没有成为默认主链路，也还没有形成真正的文件系统式产品体验。

如果目标只是“通用记忆平台，附带 file-centric 预览面”，那 AgentMem 已经在路上；
如果目标是“像 Claude Code 一样，记忆默认围绕项目文件、层级记忆文件、子树上下文、工作区作用域来组织”，那当前实现还没有达到这个级别。

一句话判断：

**AgentMem 现在的状态是“file-centric contract 已经出现，但 file-centric runtime 和 file-centric product experience 还没有真正落地”。**

## 2. 先回答用户问题：它是不是以文件为核心的记忆平台？

答案要分三层说：

### 2.1 从方向和命名上看：是

当前仓库已经明确引入了：

- `crates/agent-mem-resource`
- `crates/agent-mem-category`
- `crates/agent-mem-extraction`
- `crates/agent-mem-proactive`

而且这些模块的命名、DTO、路由、SDK 接口都在强调：

- `resource`
- `category`
- `extraction`
- `migration`
- `proactive`

这是非常典型的文件中心化平台语义，不再只是传统的 memory row CRUD。

### 2.2 从顶层产品主线看：不是

顶层 `agent-mem` 统一 API 的文档和快速开始仍然是：

- `Memory::new()`
- `mem.add(...)`
- `mem.search(...)`

见：

- `crates/agent-mem/src/lib.rs:1-80`

服务端核心公共模型开头也仍然是：

- `MemoryRequest`
- `UpdateMemoryRequest`
- `SearchRequest`

见：

- `crates/agent-mem-server/src/models.rs:1-90`

这说明平台“默认心智”依然是传统记忆平台，而不是文件中心化平台。

### 2.3 从运行时实现看：还远未真正做到

最关键的证据是：

- 顶层 `agent-mem` 虽然已经导出 file-centric DTO：
  - `crates/agent-mem/src/lib.rs:112-120`
- 但顶层 file-centric 方法大多直接返回 preview error：
  - `crates/agent-mem/src/memory.rs:1666-1760`

也就是说：

- **对外 contract 已经先行**
- **真正可用的 runtime 还没有跟上**

所以当前不能把 AgentMem 定义成“已完成的文件中心记忆平台”，只能定义成：

**正在向文件中心记忆平台演进的混合态平台。**

## 3. 外部搜索结果：OpnViking 的核心理念到底是什么

## 3.1 公开搜索结论

本轮对 `OpnViking` 的公开搜索，没有找到它对外发布的一套 AI memory product、agent memory runtime 或 Claude Code 风格记忆系统实现。

因此，`OpnViking` **不能被当作一个现成的记忆平台产品对标物**。

它更像是一个基础设施/平台工程方向的团队或品牌，其公开内容主要强调：

- full control
- trust
- sovereignty
- open standards
- no vendor lock-in
- compliance / privacy
- automation / infrastructure-as-code

也就是说，`OpnViking` 给我们的不是“记忆算法参考”，而是**平台治理和交付哲学参考**。

## 3.2 从公开材料中提炼出的可用理念

基于其公开网站和介绍页，可以合理提炼出以下几条平台理念：

1. **主权优先**：核心数据和运行路径应尽量可本地化、可自托管、可审计。
2. **开放优先**：协议、格式、接口尽量开放，不被某个私有控制面锁死。
3. **无锁定优先**：迁移路径要清晰，不能把用户记忆困在单一数据库黑箱里。
4. **合规优先**：权限、审计、数据边界、删除与保留策略需要一开始就设计进去。
5. **自动化优先**：平台状态应可声明、可复现、可部署，而不是大量人工点操作。

## 3.3 这些理念对 AgentMem 的意义

如果 AgentMem 真要做成文件中心化 AI 记忆平台，那么 `OpnViking` 给出的启发不是“怎么做记忆提取”，而是：

### 平台层必须满足

1. 记忆最好有**可读、可迁移、可版本化**的开放表示。
2. 项目级记忆不能只活在数据库里，至少要能映射到文件/目录/manifest。
3. 迁移、回滚、审计、保留策略必须是正式能力，不是文档承诺。
4. 本地模式、企业自托管模式、云模式要能共享同一套 contract，而不是三套不同真相。

所以这次更新后的一个重要判断是：

**OpnViking 更适合作为 AgentMem file-centric 平台的“治理原则参考”，而不是“功能路线参考”。**

## 4. Claude Code 类文件中心记忆平台的本质是什么

要判断 AgentMem 是否正在构建“Claude Code 类”的平台，先得把这个对象说清楚。

## 4.1 Claude Code 的官方记忆机制是文件层级，不是抽象 MemoryRow

从 Anthropic 官方 Claude Code 文档可以提炼出几个关键特征：

1. 记忆存放在实际文件中，核心载体是 `CLAUDE.md`。
2. 记忆有层级：
   - 企业策略层
   - 用户全局层
   - 项目层
3. Claude Code 会根据当前工作目录向上查找，并结合当前子树上下文决定加载哪些记忆文件。
4. 记忆可以通过文件引用和导入组合，而不是只能靠数据库检索。

这套模型的关键不是“有个 AI 记忆库”，而是：

**记忆与项目文件树、目录层级、当前工作上下文天然耦合。**

## 4.2 Claude Code 类平台至少应具备的能力

如果把“Claude Code 类文件中心平台”抽象出来，至少应满足：

1. **文件路径是一级地址空间**
   - 记忆不是只有 `memory_id`
   - 还应有 `workspace/path/module/subtree`

2. **作用域是层级化的**
   - user/global
   - project
   - subtree/module
   - session/scratch

3. **记忆默认围绕工作区组织**
   - 先知道当前在哪个 repo / 子目录
   - 再决定加载哪些记忆

4. **记忆是可读、可编辑、可版本化的**
   - 文件
   - manifest
   - git 历史
   - diff

5. **检索不是单纯向量搜索**
   - 还应包括 path-aware、scope-aware、category-aware、workspace-aware 检索

6. **后台整理是默认能力**
   - 汇总
   - 去重
   - 分类
   - 归档
   - 压缩

## 4.3 Letta/MemFS 给出的进一步启发

Letta 的 MemFS 和 context repositories 路线进一步说明：

1. 文件系统式记忆不只是“把 URI 挂载进来”；
2. 它还包括：
   - git-backed context repo
   - 分支 / worktree
   - 自动整理
   - 对多 agent 并行工作的上下文隔离

因此，判断 AgentMem 是否达到 Claude Code 类平台，不能只看它有没有 `resource` 字段，而要看：

**文件树是不是平台的默认操作面。**

## 6. 但为什么它还不是一个真正的文件中心记忆平台

## 6.1 ResourceManager 目前还是内存存储，不是正式资源控制面

最核心的阻断点之一：

- `ResourceManager` 当前底层存储就是 `Arc<RwLock<HashMap<ResourceId, Resource>>>`

见：

- `crates/agent-mem-resource/src/manager.rs:39-52`

这意味着：

1. 资源 mount 后没有正式持久化资源目录；
2. 没有稳定的资源索引；
3. 没有与 workspace file tree 做正式同步；
4. 没有版本化、变更检测、快照管理。

所以它现在更像“文件 URI 的运行时挂载器”，不是“文件中心记忆空间”。

## 6.2 CategoryManager 当前默认也是内存态

默认的类别系统是：

- `InMemoryCategoryManager`

见：

- `crates/agent-mem-category/src/manager.rs:72-126`
- `crates/agent-mem-server/src/routes/file_centric.rs:30-49`

这说明：

- 类别层级有接口和模型；
- 但还没有正式变成可持久化、可迁移、可回放的生产级目录树。

## 6.3 FileCentricState 启动时连 extraction pipeline 都没有默认装配

`FileCentricState` 初始化时：

- `extraction_pipeline: Arc<RwLock<Option<ExtractionPipeline>>>`
- 默认值是 `None`

见：

- `crates/agent-mem-server/src/routes/file_centric.rs:30-49`

这意味着 file-centric 的提取主链路并不是启动即就绪。

## 6.4 extract_resource 在 pipeline 未配置时返回“成功 stub”

这一点尤其危险。

`extract_resource()` 在 pipeline 没有配置时，并没有返回明确失败，而是返回：

- `status: Succeeded`
- `warnings: ["Extraction pipeline not configured - returning stub result"]`

见：

- `crates/agent-mem-server/src/routes/file_centric.rs:438-507`

这会导致上层误判：

- 看起来抽取成功了
- 实际什么都没抽出来

这在生产系统里是非常坏的信号，因为它制造了**假成功**。

## 6.5 conv:// 与 doc:// resolver 仍然是 placeholder

文件中心平台不只是挂载本地文件，还要能真正把“对话、文档、引用对象”统一成资源。

但当前 resolver 中：

- `conv://` 是 placeholder
- `doc://` 是 placeholder

见：

- `crates/agent-mem-resource/src/resolver.rs:200-250`

因此：

- 平台还没有统一文件、文档、会话、引用对象的真实资源层。

## 6.6 顶层 file-centric API 仍然是 preview error

这是最能说明“产品还没成”的证据。

顶层 `Memory` 上的这些方法：

- `mount_resource`
- `get_resource`
- `extract_resource`
- `list_categories`
- `search_categories`
- `plan_legacy_migration`
- `apply_legacy_migration`
- `rollback_legacy_migration`
- `list_proactive_tasks`
- `run_proactive_task`
- `cancel_proactive_task`
- `get_scheduler_stats`

都直接返回 `file_centric_preview_error(...)`

见：

- `crates/agent-mem/src/memory.rs:1666-1760`

所以从最终用户视角看，file-centric 还没有成为 `agent-mem` 的正式能力。

## 6.7 Server 仍然以 MemoryRequest/SearchRequest 为主公共模型

server 模型文件开头仍然是传统 memory CRUD：

- `MemoryRequest`
- `UpdateMemoryRequest`
- `SearchRequest`

见：

- `crates/agent-mem-server/src/models.rs:10-90`

这说明服务端的默认公共叙事没有切换到：

- project
- resource
- path
- category
- workspace scope

## 6.8 顶层文档仍然把平台描述成通用记忆 API，不是工作区文件记忆系统

顶层文档现在强调的是：

- 一行初始化
- add/search
- 多记忆类型
- 8 个 memory agents

见：

- `crates/agent-mem/src/lib.rs:1-80`

这和 Claude Code 的核心体验完全不同。

Claude Code 的关键不是“add 一条 memory”，而是：

- 当前在哪个工作区
- 该加载哪层记忆文件
- 哪些目录记忆与当前子树相关

AgentMem 现在还没有把这一层做成默认入口。

## 7. 因此，AgentMem 当前最准确的定位是什么

我认为最准确的定位是：

**AgentMem 当前是一个“legacy memory platform 正在向 file-centric platform 过渡”的混合型平台。**

更细一点说：

### 已经完成的部分

1. file-centric 语言体系已经出现
2. 资源、类别、提取、主动任务四个模块已经拆出来
3. 多语言 SDK 已开始跟随合同扩展

### 尚未完成的部分

1. 文件树没有成为默认地址空间
2. 工作区 scope 没有成为默认上下文入口
3. 资源/类别默认实现仍是内存态
4. 顶层 API 没有真正闭环
5. migration / proactive / extraction 状态机很多还是 preview 或 stub

所以它不是“没有 file-centric”，而是：

**有 file-centric 架构苗头，但还没有 file-centric 产品闭环。**

## 8. 生产级结论也需要随之更新

如果问题换成：

> AgentMem 当前是否达到生产级？

那结论仍然是：

**没有。**

而且如果把标准提高到：

> 是否达到“Claude Code 类文件中心 AI 记忆平台”的生产级？

那答案更明确：

**距离更远。**

因为除了上一版已经指出的通用生产问题之外，file-centric 路线还额外存在这些阻断：

1. 资源与类别默认不持久化
2. 提取主链路默认未装配
3. 抽取 stub 会假成功
4. 顶层 API 仍是 preview
5. 迁移与主动任务仍大量未实现
6. 没有工作区层级记忆发现机制
7. 没有像 `CLAUDE.md` 那样的层级文件记忆入口
8. 没有 git/versioning/worktree 级别的文件记忆操作面

## 9. 要把 AgentMem 真正做成 Claude Code 类文件中心平台，必须补哪条主链路

我建议 AgentMem 接下来不要再把注意力放在“继续加更多 memory type”，而要收敛到下面这条**唯一主链路**：

```text
Workspace / Repo
  -> File/Doc/Conversation Resource
  -> Extraction Pipeline
  -> Category Hierarchy
  -> Path-aware / Scope-aware Retrieval
  -> Proactive Reflection / Summarization / Cleanup
  -> Human-readable memory files + durable indexes
```

这个链路才是 Claude Code 类平台的核心。

## 10. 目标架构建议：把 file-centric 从“附加面”改成“默认面”

## 10.1 建议的正式产品形态

我建议 AgentMem 未来正式收敛成以下结构：

### 层 1：工作区记忆入口

引入类似 Claude Code 的层级记忆文件机制，例如：

- `~/.agentmem/AGENTMEM.md`：用户全局记忆
- `<repo>/AGENTMEM.md`：项目级记忆
- `<repo>/subdir/AGENTMEM.md`：子树级记忆
- `<repo>/.agentmem/resources/`：资源清单与派生结果
- `<repo>/.agentmem/categories/`：类别层级与摘要

注意，这里不是要求机械复制 `CLAUDE.md`，而是要求：

**记忆必须以真实文件树成为默认入口。**

### 层 2：资源控制面

把 ResourceManager 从 in-memory 挂载器升级成正式控制面：

- 持久化资源索引
- path / uri / hash / media_type / snapshot
- 文件变更检测
- 工作区扫描
- 外部文档引用挂载

### 层 3：类别与目录树

把 Category 从“内存树”升级成正式目录组织系统：

- 路径稳定
- 摘要稳定
- item_count 与 resource_count 一致
- 可浏览、可回滚、可导出

### 层 4：提取和派生层

Extraction pipeline 必须成为资源进入系统后的默认路径，而不是可选实验层：

- mount -> parse -> extract -> categorize -> index

而且 pipeline 未配置时必须失败，不允许假成功。

### 层 5：检索与上下文装配层

真正的 file-centric 检索应优先支持：

- 当前工作目录相关记忆
- 当前文件路径相关记忆
- 当前子树相关记忆
- 当前资源类别相关记忆
- 最近改动资源相关记忆

而不是继续只围绕：

- `MemoryType`
- `agent_id`
- `user_id`

### 层 6：主动整理层

当前 proactive crate 可以保留，但必须真正接入：

- category summary
- stale resource archive
- duplicated knowledge merge
- path-level digest
- session scratch -> project memory 提升

## 10.2 这套目标架构与 OpnViking 理念的关系

如果按 `OpnViking` 的公开理念来校验，这套架构还有几个必须满足的要求：

1. **本地主权**：本地文件可读可控
2. **开放格式**：markdown/json/manifest，而不是纯黑箱数据库
3. **可迁移**：本地、自托管、云模式之间可平滑迁移
4. **可审计**：谁改了什么记忆、何时提取、何时归档
5. **自动化**：可通过 CI/脚本/策略批量管理记忆状态

这正是 AgentMem 目前最缺的一层产品纪律。

## 11. 更新后的分阶段改造计划

## Phase A：重新定义平台真相

目标：先让代码、文档、产品表面说同一种语言。

### 任务

1. README、网站、SDK 文档明确区分：
   - legacy memory surface
   - file-centric preview surface
2. 删除或降级“已是企业级文件中心平台”的叙事。
3. 把顶层主叙事从“memory types”改成“双轨清晰说明”：
   - 现有正式面
   - 正在建设的 file-centric 面

### 验收

- 用户不会再误以为 file-centric 已经 fully GA。

## Phase B：建立真正的文件中心基座

目标：让 file-centric 不再只是一层 API DTO。

### 任务

## 12. 最终判断

更新后的最终判断如下：

### 12.1 关于 OpnViking

`OpnViking` 不是一个已公开的 AI 记忆平台对标物；
它给 AgentMem 的价值在于：

- 主权
- 开放
- 无锁定
- 合规
- 自动化

这些是**平台治理原则**，不是直接功能蓝图。

### 12.2 关于 AgentMem

AgentMem 当前：

- **在意图上**：正在走向文件中心化
- **在合同层**：已经出现 file-centric baseline
- **在实现层**：仍然是 legacy memory 平台主导
- **在产品层**：还不是 Claude Code 类文件中心记忆系统
- **在生产级上**：距离正式可交付仍有明显差距

### 12.3 最重要的一句话

**AgentMem 现在不是“已经完成的 Claude Code 类文件中心 AI 记忆平台”，而是“拥有正确方向、但仍卡在 contract-first / runtime-incomplete 阶段的过渡平台”。**

接下来最重要的不是继续加更多记忆功能，而是：

**让文件树、工作区作用域、层级记忆文件、资源控制面、提取链路和主动整理，真正成为平台默认主线。**
